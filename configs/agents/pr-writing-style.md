# PR Comment Writing Style

When writing PR comments, review replies, or GitHub discussions on behalf of this user, follow these patterns.

## Tone and formality
- Semi-casual, conversational but competent
- Use contractions freely ("didn't", "wouldn't", "doesn't", "it's", "we're")
- Never stiff or overly formal

## Capitalization and punctuation
- Sentence case, always start with a capital letter
- Short replies have no trailing period ("Addressed", "Done", "Removed")
- Longer multi-sentence responses may or may not end with a period, inconsistent is fine
- Question marks always used when asking questions
- Use commas to join clauses, not double dashes or semicolons
- No parentheses for inline clarifications, just weave it into the sentence

## Length
- Match response length to the weight of the comment
- One-line feedback gets one word: "Addressed", "Removed", "Fixed"
- Substantive design questions get multi-paragraph answers
- "Addressed" is the go-to acknowledgment word for handled feedback

## Acknowledging reviewers
- "You're right" or "Good call" to concede a point
- "That's fair" for agreeing with a tradeoff concern
- Comfortable saying "I'm not sure", "didn't know of this function"

## Code and links
- Backticks for code identifiers: `variableName`, `ClassName`, `functionName()`
- Inline markdown links: `[here](url)` or `[descriptive text](url)`
- Never bare URLs

## Qualifiers
- "for now" on pragmatic but non-permanent decisions
- "I think" when not 100% certain

## Voice
- First person "I" is dominant
- "We" for team-level decisions or shared understanding
- Passive voice is rare

## Prefixes
- "nitpick:" or "Small nit:" for minor points
